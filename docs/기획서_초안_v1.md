## 1. 추진 배경

### 1-1. 문제 — 발음 피드백은 '사람'이 있어야만 가능했다
발음은 혼자 연습하기 가장 어려운 영역이다. 단어와 문법은 책과 앱으로 배울 수 있지만, **내 발음이 맞는지 틀린지는 들어줄 사람이 없으면 알 수 없다.** 기존 한국어 학습 앱은 듣기·어휘 중심이고, 발음 평가를 제공하는 서비스도 문장 전체의 합/불 판정에 그쳐 **"어디가, 왜 틀렸는지"** 를 알려주지 않는다.

### 1-2. 기술적 기회 — STT가 충분히 좋아졌다
Whisper 등 오픈소스 STT의 한국어 인식률이 실용 수준에 도달했고, 로컬에서 무료로 돌릴 수 있다. 사용자 발화를 텍스트로 받아 **정답 문장과 자모(초성·중성·종성) 단위로 정렬·비교**하면, 사람 없이도 "받침 ㄹ이 ㄴ으로 발음됐다" 수준의 구체적 피드백이 가능하다.

### 1-3. 개인 목표와의 연결
- **포트폴리오**: STT·TTS·ML을 장식이 아닌 핵심 기능으로 쓰는 프로젝트. "규칙 기반 베이스라인 → ML 전환으로 개선"이라는 서사를 만들 수 있다.
- **코어 자산**: 녹음 → STT → 분석 파이프라인은 음성 기반 해커톤 아이템(감정 일기, 시니어 말벗 등)에 그대로 재사용된다.
- **데이터 수집 용이성**: 본인 녹음으로 즉시 테스트 데이터를 만들 수 있고, AI Hub 공개 데이터(KsponSpeech 등)로 보강 가능. 서비스 사용 자체가 학습 데이터를 쌓는 구조로 설계한다.

## 2. 서비스 목표 및 범위

### 2-1. MVP — 핵심 루프 하나를 완성한다
> **화면에 문장 제시 → TTS 표준 발음 재생 → 사용자 녹음 → STT 인식 → 자모 단위 비교 → 점수 + 틀린 부분 하이라이트**

이 루프가 돌아가면 서비스가 성립한다. 회원가입·랭킹·소셜 기능은 8주 범위에서 **의도적으로 제외**한다.

### 2-2. 개발 범위
| 구분 | 내용 | 상태 |
|---|---|---|
| 발음 비교 엔진 | 정답 문장 vs STT 결과, 자모 단위 정렬·편집거리 점수화 | **MVP (핵심)** |
| 핵심 루프 UI | 브라우저 녹음(MediaRecorder) → 점수·하이라이트 표시 | MVP |
| TTS 표준 발음 | 문장별 표준 발음 음원 생성·캐싱 | MVP |
| 문장 세트 | 난이도별 연습 문장 30개 (받침·연음·경음화 등 유형별) | 5주차 |
| 학습 기록 | 시도 저장, 점수 추이 그래프, 취약 발음 통계 | 6주차 |
| ML 오류 패턴 분석 | 자주 틀리는 음소 패턴 분류, 사용자별 취약 발음 예측 | 7주차 |
| 배포 | 클라우드 배포 + 데모 영상 | 8주차 (선택) |

## 3. 주요 기능 및 핵심 기술

### 3-1. 발음 비교 엔진 (프로젝트의 심장)
- 한글을 **자모 단위로 분해**(`jamo` 라이브러리)한 뒤, 정답과 인식 결과를 정렬(alignment)해 편집거리 기반 유사도를 계산한다.
- 음절 단위 비교로는 "밥→받"을 통째로 틀렸다고만 판정하지만, 자모 단위로는 **"종성 ㅂ→ㄷ 오류"** 로 특정할 수 있다 — 이것이 피드백 품질의 차이를 만든다.
- 표준 발음 규칙(연음, 경음화, 비음화 등)을 반영해 **표기 기준이 아닌 발음 기준 정답**과 비교한다. 예: "국물"의 정답 발음은 [궁물].
- 독립 모듈로 분리해 백엔드와 결합도를 없앤다 → 해커톤 재사용 자산.

### 3-2. ML 전략 — 3단계로 리스크를 나눈다
| 단계 | 내용 | 시점 |
|---|---|---|
| ① 규칙 기반 베이스라인 | 자모 편집거리 점수화. ML 없이 데모가 돌아가는 상태 확보 | 2주차 |
| ② 데이터 축적 | "정답 문장 + 발화 녹음 + STT 결과" 트리플이 사용 중 자동으로 쌓이는 구조 | 3~6주차 |
| ③ ML 적용 | 축적 데이터 + AI Hub 데이터로 오류 패턴 분류·취약 발음 예측 모델. 베이스라인 대비 정량 비교 | 7주차 |

> 포인트: ML이 실패해도 ①이 있어 서비스는 성립하고, ML이 성공하면 "베이스라인 대비 X% 개선"이라는 포트폴리오 서사가 생긴다.

### 3-3. 활용 기술
| 구분 | 선택 | 비고 |
|---|---|---|
| STT | OpenAI Whisper 로컬 `small`~`medium` | 무료·오프라인 개발. CPU가 느리면 Whisper API 전환 |
| TTS | 네이버 클로바 보이스 API (무료 대안: Edge-TTS) | 한국어 자연스러움 우선. 음원은 캐싱해 호출 최소화 |
| 발음 비교 | `jamo` + 편집거리(Levenshtein) 정렬 | 자체 구현 핵심 모듈 |
| ML | scikit-learn (심화: wav2vec2 계열 파인튜닝) | ③단계에서 적용 |
| 백엔드 | FastAPI + SQLite | 가볍게. 배포 시점에 필요하면 교체 |
| 프론트 | HTML+JS 또는 React, MediaRecorder API | 웹 우선 — 데모 공유가 쉽다 |
| 데이터 | 본인 녹음 + AI Hub KsponSpeech 등 | [TODO: AI Hub 데이터 신청·약관 확인] |

## 4. 시스템 아키텍처

```
[브라우저]
  녹음(MediaRecorder) ──audio──▶ [FastAPI]
  문장/점수 표시 ◀──JSON──        ├─ Whisper STT
  TTS 음원 재생 ◀──mp3──          ├─ 발음 비교 엔진 (자모 정렬 + 점수화) ★독립 모듈
                                  ├─ TTS 모듈 (클로바/Edge-TTS, 음원 캐싱)
                                  ├─ ML 분석기 (오류 패턴 분류) — 7주차 합류
                                  └─ SQLite (문장, 시도 기록, 점수)
```

## 8. 기대 효과

1. **포트폴리오** — STT·TTS·ML이 핵심 기능인 완결된 서비스 + "규칙 기반 → ML 개선" 정량 서사.
2. **재사용 자산** — 녹음→STT→분석 파이프라인과 발음 비교 엔진은 음성 기반 해커톤 아이템에 즉시 이식 가능.
3. **확장 여지** — 외국인 한국어 학습자 특화, 아나운서·발표 준비생 모드 등 타겟 전환이 쉬운 구조.
