## 1주차 — 기술 검증 (≈12h)

이 주의 목적은 코드가 아니라 **판단**이다. 최대 리스크(Whisper의 자동 교정)를 여기서 확인하고 STT 방향을 결정한다.

### 작업
- [ ] **환경 세팅** (2h): Python 가상환경, `openai-whisper`, `ffmpeg`, `edge-tts` 설치. `experiments/` 폴더 생성.
- [ ] **STT 검증 스크립트** (3h): 오디오 파일 → Whisper 인식 텍스트 출력. `small` / `medium` 모델별 결과·처리시간 비교.
- [ ] **핵심 실험 — 교정 문제** (4h):
  - 정상 발음 문장 10개 + **의도적으로 틀리게 발음한 문장 10개** 직접 녹음 (예: "국물"을 [국물]로 또박또박, "밥"을 [받]으로).
  - 각 녹음을 Whisper에 넣고 **틀린 발음이 틀린 대로 인식되는지** 기록.
  - 완화 실험: `temperature` 조정, `initial_prompt` 제거, `small` vs `medium` 비교.
  - 결과를 `experiments/whisper_교정실험.md`에 표로 기록.
- [ ] **TTS 검증 스크립트** (2h): 문장 → Edge-TTS mp3 생성. 한국어 음질 확인. (클로바 API는 품질이 아쉬울 때만 5주차에 검토 — 초기엔 무료로.)
- [ ] 선택 (1h): `g2pk`(표기→표준발음 변환 라이브러리) 설치해 "국물→궁물" 변환 확인. 2주차 준비.

### 완료 기준
- 교정 실험 결과표가 존재하고, 아래 **결정**이 내려져 있다.

### 결정 포인트 (이 주에 반드시 결정)
| 실험 결과 | 결정 |
|---|---|
| 틀린 발음 20개 중 대략 절반 이상이 틀린 대로 인식됨 | Whisper `small`로 진행 (계획대로) |
| 대부분 교정되어 평가 불가 | 2주차 전반부에 wav2vec2 계열 음소 인식(예: 한국어 파인튜닝 모델) 검증을 끼워 넣고, 5주차 UX 작업을 축소해 시간 확보 |

> **✅ 결정 완료 (2026-07-04) — 부분 발동, 하이브리드로 진행.**
> 자모 대체형 오류는 발음형 기준 4/7 보존 → Whisper `small`(t=0.0) 유지하되 비교는 발음형 공간에서.
> 발음규칙 미적용형(비음화·경음화 등)은 표기 수렴 문제로 0/3 — 텍스트 STT로는 구조적으로 감지 불가
> → **2주차 전반부에 wav2vec2 음소 인식 검증 추가, 5주차 UX 축소.** g2pk 확보는 선택→필수로 격상.
> 상세: [../experiments/whisper_교정실험.md](../experiments/whisper_교정실험.md) 3~4번 섹션.

## 2주차 — 발음 비교 엔진 v1 + wav2vec2 검증 (≈12h)

프로젝트의 심장. 이 모듈은 FastAPI 없이 **순수 Python 패키지**로 만든다 (해커톤 재사용 자산).
1주차 결정에 따라 전반부에 wav2vec2 음소 인식 검증이 추가됐다 (시간은 5주차 UX 축소로 확보).

### 작업 (1주차 결정으로 추가)
- [x] **g2pk 확보** (1h): MSVC C++ 워크로드 + mecab-ko-msvc(C:\mecab) + libmecab.dll 복사로 해결 (2026-07-04, 절차: `experiments/README.md`). `g2p_check.py` 8/8 규칙 변환 일치.
- [x] **wav2vec2 음소 인식 검증** (3h): `kresnik/wav2vec2-large-xlsr-korean`을 1주차 녹음 20개로 검증 (`experiments/wav2vec2_check.py`).

> **✅ wav2vec2 검증 결과 (2026-07-04)** — 상세: `experiments/results/wav2vec2_kresnik_wav2vec2-large-xlsr-korean.md`
> - **비음화 미적용 감지 성공**: 정상 "국물이" vs 또박또박 "국무리"로 구분 출력 — Whisper가 0/3이던 영역에서 첫 성과.
>   정상/오발음 쌍 20개 중 9/10에서 출력이 달랐다. CPU 문장당 1.1~2.4초로 속도도 합격.
> - **유음화·경음화는 미감지** (모델도 표기 학습이라 표기 수렴의 영향을 받음). 규칙 미적용형 감지 1/3.
> - **기본 인식률 낮음**: 정상 발음 10개 중 5개 오인식 → 이대로 점수화에 쓰면 정상 발화에 억울한 감점.
> - **결론**: MVP(3~4주차)는 Whisper 단독 경로로 간다. wav2vec2는 7주차에 '해당 문장의 규칙 위치만 조준 검사'하는
>   보조 신호로 재검토 (전역 점수화가 아니라 특정 음절 구간만 보면 오인식 노이즈 영향이 줄어든다).
>   IPA 음소 모델(`facebook/wav2vec2-lv-60-espeak-cv-ft`)은 espeak-ng 네이티브 의존성 때문에 보류 — 7주차 후보.

### 작업 (기존)
- [ ] **표준 발음 변환** (2h): `g2pk`로 표기 문장 → 발음 문장 변환 ("국물이 좋다" → "궁무리 조타"). 비교 기준은 항상 이 발음형.
- [ ] **자모 분해** (2h): `jamo` 라이브러리로 발음형·인식형을 초/중/종성 시퀀스로 분해. 종성 없음도 명시적 토큰으로 처리.
- [ ] **정렬 알고리즘** (4h): 두 자모 시퀀스를 정렬(difflib `SequenceMatcher` 또는 직접 구현한 편집거리 역추적). 삽입/삭제/대체를 자모 단위로 산출.
- [ ] **점수화 + 오류 리포트** (2h): 전체 점수(0~100) + 오류 목록(위치, 기대 자모, 실제 자모, 초/중/종성 구분) 반환. 출력 형태는 dict — 그대로 API 응답이 된다.
- [ ] **단위 테스트** (2h): pytest로 20케이스 이상. 정상 일치, 종성 오류, 연음 문장, 완전 불일치, 빈 입력 등.

### 완료 기준
- `engine.compare(정답문장, 인식문장)` 한 번 호출로 점수와 오류 목록이 나오고, 테스트가 전부 통과한다.
- 1주차 녹음 20개를 엔진에 통과시켜 점수가 직관과 대체로 일치함을 확인했다.

### 결정 포인트
- API 응답 JSON 스펙을 이 주에 확정하고 문서화한다(`docs/api_spec.md`). 이후 프론트·백엔드는 이 스펙만 본다.

## 3주차 — 백엔드 파이프라인 (≈12h)

### 작업
- [ ] **FastAPI 프로젝트 구조** (2h): `app/`(API) · `engine/`(비교 엔진) · `stt/` · `tts/` 분리. SQLite 스키마: `sentences`(문장, 발음형, 난이도, 유형), `attempts`(문장 id, 녹음 파일 경로, STT 결과, 점수, 오류 JSON, 시각).
- [ ] **문장 API** (1h): `GET /sentences` — 초기엔 하드코딩 문장 10개 시드.
- [ ] **TTS API** (2h): `GET /sentences/{id}/audio` — 최초 호출 시 Edge-TTS 생성 후 파일 캐싱, 이후 캐시 반환.
- [ ] **평가 API** (5h): `POST /attempts` — 오디오 업로드 → ffmpeg으로 wav 변환 → Whisper 인식 → 비교 엔진 → 점수 JSON 응답 + DB 저장. **녹음 원본 파일도 보관** (7주차 ML 데이터).
- [ ] **수동 테스트** (2h): curl/HTTPie로 1주차 녹음 파일들을 업로드해 전 구간 확인. 처리 시간 측정(목표: 문장당 10초 이내, CPU 기준).

### 완료 기준
- 오디오 파일 하나를 POST하면 점수 JSON이 돌아오고 DB에 기록이 남는다.

### 리스크 체크
- CPU 추론이 10초를 크게 넘으면: 모델을 `base`로 낮춰 재실험 → 그래도 느리면 Whisper API 전환을 4주차 시작 전에 결정.

- [ ] **통합 디버깅** (3h): 브라우저 오디오 포맷(webm/ogg) → 서버 ffmpeg 변환 확인, 마이크 권한 처리, 로딩 표시.

### 완료 기준 = MVP 체크포인트

- 30초 데모 영상이 존재한다.

## 5주차 — 문장 세트 + UX 최소 다듬기 (≈10h)

1인 기준으로 UX는 **최소한만**. 이 주의 무게중심은 문장 세트(콘텐츠)다.
**1주차 결정에 따라 이 주가 축소 1순위** — 2주차 wav2vec2 검증에 밀린 작업이 있으면 여기의 선택 작업(문장 목록 화면)과 하이라이트 개선을 먼저 버린다.

### 작업
- [ ] **유형별 문장 세트 30개** (4h): 발음 규칙 유형별로 설계 — 연음(6), 경음화(5), 비음화(5), 격음화(4), 받침 대표음(5), 종합(5). 각 문장에 유형 태그·난이도 저장. `g2pk` 변환 결과를 눈으로 검수(전체 30개면 검수 가능한 양).
- [ ] **하이라이트 개선** (2h): 자모 오류를 음절 단위로 묶어 표시 + 툴팁으로 "기대: ㅂ / 발음됨: ㄷ".
- [ ] **에러 처리** (3h): 무음·1초 미만 녹음 거부, STT 빈 결과 안내, 업로드 실패 재시도.
- [ ] 선택 (1h): 문장 목록 화면(유형·난이도 필터).

### 완료 기준
- 30문장이 유형 태그와 함께 DB에 있고, 이상 입력에서 앱이 죽지 않는다.

## 운영 원칙 (1인 프로젝트용)

1. **주 단위 리듬**: 주 시작에 그 주 체크리스트 확인 → 주말에 완료 기준 점검. 세션 시작 시 이 문서를 기준으로 진행 상황을 물어보면 된다.
2. **밀리면 버린다**: 필수/선택이 나뉘어 있다. 밀릴 때 일정을 늘리지 말고 선택 작업을 버린다.
3. **매주 커밋 + 짧은 기록**: `docs/log.md`에 주당 3줄(한 것 / 막힌 것 / 다음). 회고 글의 재료가 된다.
4. **해커톤 충돌 주간**: 해당 주는 필수 작업 중에서도 최소 1개만 수행하고 로드맵을 1주 밀되, MVP(4주차)와 ML(7주차)은 사수한다.
